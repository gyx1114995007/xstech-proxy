### 核心修复：xstech 带文件请求格式对齐

根据用户提供的 `api-docs/抓包.txt` 与 xstech 前端真实行为，确认 `/api/chat/completions` 带文件请求体中并不包含：

```text
useImages
useFiles
```

中转站此前主动注入 `useImages/useFiles`，会使 `deepseek-v4-flash` 上传图片时误触发上游不稳定视觉链路，表现为 `524` / `ETIMEDOUT`。修复后默认请求体对齐 xstech 前端：

```json
{
  "text": "...",
  "sessionId": "...",
  "files": [],
  "thinking": false,
  "webSearch": false
}
```

实现变更：

```text
启动/配置.js
- 新增 xstech.发送文件开关字段
- 环境变量：XSTECH_SEND_FILE_FLAGS
- 默认 false

服务层/请求转发.js
- 默认 body 不再包含 useImages/useFiles
- 仅 XSTECH_SEND_FILE_FLAGS=true 时恢复旧字段
```

验证结果：

```text
deepseek-v4-flash 带图请求已可正常回复
行为与 xstech 前端一致：可上传并回复，但该模型通常不识别图片内容
不再出现本问题相关的 524 / ETIMEDOUT
```

1. xstech 上游网络不稳定时，面板可能显示 `ETIMEDOUT / ECONNABORTED / AggregateError`，应优先使用“上游网络检测”。

3. 备份恢复必须显式填写要恢复的文件名，服务端会先为原文件生成 `.restore-backup-*`。
4. 数据文件健康检查中的可选文件缺失不算异常；JSON/JSONL 解析失败才会标记 bad。
5. 误判词在线保存会自动备份旧 `误判词.json` 并立即重载规则。
6. 每次修改核心路由、面板或运维工具后，建议执行 `npm run smoke`，快速确认 `/health`、`/panel`、关键 `/debug/*` 和 `/v1/models` 仍可用。

## 关键设计决策

### 1. 请求改写策略
- 当前采用 `xs-relay` 方案 A：`nonce 工具提示词 + 分隔线 + remaining request JSON`
- **核心原则：下游请求体只抽离中转站自身需要消费的字段，其余一点都不要改变**
- 已消费字段包括：`model`、`stream`、`temperature`、`top_p`、`max_tokens`、`max_completion_tokens`、`presence_penalty`、`frequency_penalty`
- `messages`、`tools`、`tool_choice`、自定义字段等必须原样保留在 remaining JSON 中
- nonce 只用于提示词工程与流式解析，不写入 remaining JSON
- 最终整体作为 xstech 的 `text` 字段发送

### 2. 会话池策略
- 会话按模型分池
- 会话归还时**保持当前模型不变**
- 下游没传参数时，归还只解锁，不做任何重置
- 下游传了 `temperature` / `presence_penalty` / `frequency_penalty` 时，归还后恢复默认参数
- `contextCount` 固定保持 `0`，不主动走 `context-clear`

### 3. 误判词规避策略
- 规避方式：在误判词字符间插入零宽空格 `U+200B`
- 误判词记录在 `误判词.json`
- 下次请求先预替换，0 额外请求
- 只把真正定位到的误判片段写入规则，不应该把整句长文本写进去

### 4. 误判检测触发条件
- **唯一触发条件：** `code=1` 且 `err` 包含 `您的提交内容包含不允许的文本`
- 其他 `code=1` 一律不当作误判
- 例如 `上游异常提示(HttpCode:429)` 不属于误判词

### 5. 流式重试目标
- 不做假流，不缓存整段回答后再一次性发给下游
- 目标是保持 SSE 真流式
- 一旦第一块命中误判，则不把错误输出到下游，后台修复并重试
- 修复成功后的第二次请求，正常真流式输出给下游

### 6. tool_calls 方案 A 设计
- 参考 `xs-relay` 的 nonce 工具调用方案
- 上游提示词要求模型在需要调用工具时输出 nonce 伪 XML 标记
- 格式核心为：开始标签包含本轮 nonce，内容为 JSON 参数对象
- 实测模型有时会把关闭标签简写为普通关闭形式，因此 parser 在已进入当前 nonce tool 模式后兼容普通关闭标签
- 安全边界仍在开始标签：只有匹配本轮 nonce 的 open tag 才能进入 tool 模式
- `流式转换.js` 会把工具调用片段转换成 OpenAI `delta.tool_calls`，最终 `finish_reason` 应为 `tool_calls`

## 误判检测当前实现

### 当前流程
1. 首次请求发送到 xstech
2. 若第一块返回 `code=1 + 不允许的文本`
3. 触发 `工具/误判检测.js`
4. 误判检测使用独立会话并发探测，不污染主会话
5. 检测过程：
   - 第1级：重叠分段
   - 第2级：细分段
   - 第3级：句级
   - 第4级：词级
6. 找到误判词后：
   - 写入 `误判词.json`
   - 对原始 text 做零宽空格规避
   - 主请求静默重试

### 词级检测当前版本
- 中文会按单字切分
- 英文数字连续块会合并成 token
- 词级不是简单按标点 split，而是做 **双字组合探测**
- 例如希望能把 `色b` 精确探测出来，而不是把整句 `角色B作为冒险者` 误当成规则

## 关键架构原则

### 模型 SSE 错误标准化与统计
- 新增 `工具/模型流错误分类.js`
- 新增 `工具/运行指标.js`
- `工具/流式转换.js` 在 xstech SSE 返回 `code !== 0` 时：
  - 将聊天流错误归类为模型流错误
  - 转成 OpenAI/NewAPI 兼容 error
  - 记录到运行指标
  - 不触发 token 刷新
  - 不归因账号
- 当前已支持分类：
  - `content_censor`：内容误判，例如 `不允许的文本`
  - `model_rate_limit`：聊天流中的模型限流，例如 SSE `code=429`、`请求过于频繁`
  - `model_timeout`：模型/供应商超时
  - `model_context_error`：上下文长度/上下文超限
  - `model_quota_error`：额度/余额/billing 问题
  - `model_no_channel`：模型无可用渠道，例如 `无可用渠道（distributor）`
  - `model_provider_auth_error`：模型供应商/渠道认证错误，例如聊天流中的 `上游异常提示(HttpCode:401)：0 - Invalid API Key`
  - `model_provider_subscription_error`：模型供应商/渠道订阅、套餐或会员错误，例如 `CodingPlan subscription expired`
  - `model_not_found_or_no_access`：模型不存在或渠道无访问权限，例如 `The model ... does not exist or you do not have access to it`
  - `upstream_model_error`：模型上游/provider 通用错误
  - `model_stream_error`：未知模型流错误
- 真实样例验证：
  - 原生 xstech 请求：
    - `POST https://xstech.one/api/chat/completions`
    - `text=1`
    - `sessionId=96679`
  - HTTP 请求成功，SSE 返回：
    - `code=1`
    - `err=上游异常提示(HttpCode:503)：0 - 分组 default 下模型 glm-5 无可用渠道（distributor）`
  - 已确认该错误属于聊天 SSE 层模型调用错误
  - 分类结果为：
    - `source=model_stream`
    - `type=model_no_channel`
    - `code=model_no_channel`
    - `action=return_error`
  - 该类错误不刷新 token，不归因账号
- 真实样例补充：
  - `上游异常提示(HttpCode:401)：0 - Invalid API Key`
  - 已归类为 `model_provider_auth_error`
  - 表示 xstech 上游模型供应商/渠道/distributor 的 API Key 无效
  - `上游异常提示(HttpCode:400)：0 - CodingPlan subscription expired`
  - 已归类为 `model_provider_subscription_error`
  - 表示 xstech 上游模型供应商/渠道账号订阅、套餐或会员过期/无效
  - `上游异常提示(HttpCode:404)：0 - The model ... does not exist or you do not have access to it`
  - 已归类为 `model_not_found_or_no_access`
  - 表示模型不存在或该上游渠道账号没有访问权限
  - 这些错误虽然包含 `HttpCode:400/401/404`，但出现在聊天 SSE 流中，因此属于模型调用层错误
  - 不触发 xs 中转站 token 刷新
  - 不归因 xstech 账号
- 下游错误脱敏原则：
  - xstech 原始模型流错误只在内部保留
  - 下游 NewAPI/客户端不直接看到完整上游错误
  - 下游只接收脱敏后的标准错误提示
  - OpenAI error 中保留：
    - `message`：安全提示

- 自动处理验证：

### 错误边界原则
- 当前项目作为 NewAPI 的 xstech 上游适配器，不承担 NewAPI 的多租户、计费、用户级限流职责
- xstech 平台错误通过 HTTP/axios 层返回：
  - 登录失败
  - token 失效
  - 认证/鉴权失败
  - session API 失败
  - 模型列表接口失败
  - 云端会话同步接口失败
- HTTP/axios 层错误由 `账号池.带Token重试()`、`请求转发.js`、会话池/同步层处理
- token 生命周期只处理 HTTP 层平台认证问题：
  - token 不从聊天流里判断
  - token 失效不属于模型调用错误
  - 只有 HTTP 层认证/token 错误才触发 token 刷新
- 模型调用错误通过聊天 SSE 流返回：
  - HTTP 200 不代表模型调用成功
  - SSE 中的 `code/err` 才表达模型调用阶段错误
  - 例如 `code=1 + err=不允许的文本`
  - 例如模型繁忙、上游 provider 错误、模型超时、模型限流等
- 聊天 SSE 流错误由 `流式转换.js` / `路由/对话补全.js` 处理
- 聊天流错误不触发 token 刷新
- 聊天流错误不归因 xstech 账号
- `不允许的文本` 是聊天流中的内容误判，继续走误判检测与 `误判词.json` 规则沉淀
- 后续成熟化方向应是：
  - 模型 SSE 错误标准化
  - 模型错误分类统计
  - OpenAI/NewAPI 兼容错误格式
  - 而不是把模型调用错误归因为账号问题

  - 当前仍处于高强度联调期，详细日志对定位问题仍然必要
  - `forward.log` 当前普通文本可能同时记录 `toolText` 与 `delta`
  - 长上下文 tool 请求会产生大量日志
  - 曾出现过 native 层 `double free or corruption`，需继续观察是否与高频日志写入有关
- 后续 P4 方向不是直接删除日志，而是增加调试开关：
  - 是否开启 raw 日志
  - 是否开启 forward delta 日志
  - 是否开启 censor 探测详细日志
  - 是否开启 tool_call 详细解析日志
  - 是否开启请求体详细日志
- 流程节点日志必须保留：
  - 服务启动/停止
  - 账号 token 恢复/刷新/失效重试
  - 模型映射刷新
  - 云端会话同步
  - 会话分配/归还/扩容
  - tool_call start/delta/done
  - 误判检测开始/命中/规则写入
  - 请求异常与重试
- 目标是“可开关的调试日志 + 永久保留流程节点日志”，而不是完全静默
